# Module 08 dry run: Building and Proving It — Integration, Evaluation & Optimization

**Date:** 2026-07-17.
**Purpose:** validate that Module 08's deterministic tier (`scripts/verify_module_08.py`) discriminates real attempts on both of its deliverables — a diagnose-and-fix defect in `src/doc_qa.py`'s `refresh_index`, and a build-from-stub evaluation harness in `src/evaluation.py` — and that it correctly chains `check_module_07` as Foundry's own cumulative-gate convention requires.

## The scenario, and why it needs to exist

Module 07 taught recognizing when *not* to reach for an agentic or retrieval-heavy pattern. Module 08 teaches the complementary skill: recognizing when retrieval-augmented generation genuinely *is* the right call, and then building and evaluating it competently. The requesting team this time is the **Platform Docs team**: employees ask ad hoc questions about internal engineering docs (runbooks, onboarding guides), the corpus is large and genuinely changes over time, and answers must be grounded in whichever version of a document is *current* — a real RAG shape, unlike Module 07's single-turn classification.

**Two required deliverables, matching the module README's own stated gate:**

1. `src/doc_qa.py` ships mostly working (chunking, indexing, retrieval, grounded answer generation all correct as shipped) — matching Module 06's diagnose-and-fix format. `refresh_index` has one seeded defect: it checks whether a `doc_id` is *present* in the index before re-chunking, never whether its *content* has actually changed. A revised runbook with the same `doc_id` keeps its stale chunks forever — directly anchored to the CCAR-P exam guide's own Sample Question 3 ("RAG returning confident-but-wrong answers after a document refresh").
2. `src/evaluation.py` ships as a stub (`evaluate`/`compare_top_k` both raise `NotImplementedError`) — matching Module 07's build-from-stub format. `evaluate` scores a labeled Q&A dataset against `doc_qa.py`, and `compare_top_k` runs the same harness under two or more `top_k` configurations as a real A/B comparison.

## Attempts constructed

1. **`correct-attempt/`** — `refresh_index` fixed (compares a content hash, not just presence); `evaluate`/`compare_top_k` fully implemented.
2. **`unfixed-attempt/`** — the shipped state exactly: staleness bug present, evaluation harness still a stub. Represents a learner who hasn't started.
3. **`stale-fix-only-attempt/`** — `refresh_index` fixed; evaluation harness still a stub.
4. **`eval-only-attempt/`** — evaluation harness fully implemented; `refresh_index` bug NOT fixed. Proves the two deliverables are genuinely independent, the same discipline Module 06's `fix-bug1-only`/`fix-bug2-only` attempts established.
5. **`broken-eval-metric-attempt/`** — both pieces "look done" (`refresh_index` fixed, `evaluate`/`compare_top_k` implemented and running), but `evaluate` has a real, plausible metric-design flaw: it counts a case correct whenever the model cited *any* document, not specifically the one the case expects — a too-lenient metric that would silently overstate real accuracy, and specifically defeats the module's own `compare_top_k` A/B test (a wrong-document citation "passes" regardless of `top_k`).

## Doubt-driven-development review, before treating this module as done

Standing practice, continued from every prior module — the third module in a row (after 06 and 07) where the review found real, live-confirmable gaps, not just style nits.

**Stage 1 (fresh Claude subagent):** 8 findings, 3 empirically confirmed by constructing submissions that passed all 21 original tests while violating the contract: a `refresh_index` doing a full rebuild every call (never checking a content hash at all) passed, because the "leaves unchanged docs untouched" test only checked VALUE equality on a frozen `Chunk` dataclass — a freshly re-derived, identical-valued chunk is still value-equal to the original, so the test couldn't distinguish "skipped the work" from "redid the work and got the same answer." A `refresh_index` that mutates its input `index` in place and returns that same object also passed, since the relevant tests compared the *returned* index against the original *after* the call — tautological when they're literally the same object. `evaluate`'s documented default `top_k=3` was never exercised by any test.

**Stage 2 (Codex cross-model):** found 4 further real issues — `evaluate` was never actually forced to route through `answer_question`'s real retrieval/context-construction path (a broken `evaluate` calling `model_client` directly with just the raw query passed every original test, since none of them checked what context the model was actually shown); `compare_top_k` was never forced to evaluate the *full* dataset per `k` value (a version silently evaluating only `dataset[:1]` for every k passed, since every original compare test used a 1-case dataset); `refresh_index`'s shallow dict copy shared mutable `List[Chunk]` object references between the old and new index for unchanged/preserved docs, so mutating one index's list would silently corrupt the other; `answer_question`'s malformed-response check validated field *presence* but not *type* — a response like `{"cited_doc_ids": "runbook"}` (a bare string, not a list) passed and silently produced character-by-character citations via `list("runbook")`.

One Codex claim was tested live and **refuted**: that the checker's `f"{expected} passed" in output` check could be bypassed by a submission printing a fake `"21 passed"` line before calling `os._exit(0)`. Constructed and ran this exact submission against the real checker — it correctly reports `FAIL`. Root cause: pytest's own internal output capturing during test collection buffers the submission's `print()` call, and `os._exit(0)` terminates the process without flushing that buffer to the real stdout `subprocess.run` is capturing — even with an explicit `sys.stdout.flush()` added to the submission. Verified, not assumed; recorded as a refuted finding rather than silently dropped.

**Fixed:** `refresh_index` now copies each doc_id's chunk list, not just the outer dict (`{doc_id: list(chunks) for doc_id, chunks in index.chunks.items()}`), closing the shared-mutable-list issue while still skipping unnecessary re-chunking for unchanged docs. `answer_question` now validates `isinstance(raw["answer"], str)` and `isinstance(raw["cited_doc_ids"], list)`, fixed in both the reference implementation and the shipped stub (a base-design fix, not part of the seeded pedagogical defect, following Module 06's precedent for uniform base-design fixes). Test suite grew from 21 to 32, with each of the 5 subtlest new tests verified live against a constructed counter-implementation matching the exact bug it targets before being trusted: a `chunk_document`-spying test (via `unittest.mock.patch`) proving unchanged docs aren't re-chunked; a before/after snapshot test proving the original index object isn't mutated; a list-identity mutation test proving returned chunk lists are independent of the original's; a `GroundedScriptedModelClient` test double (cites only doc_ids actually present in the context it was shown) used to prove `evaluate` genuinely routes through real retrieval rather than talking to `model_client` directly; a two-case dataset proving `compare_top_k` evaluates the full dataset per k, not a subset. Plus default-`top_k`, default-`chunk_size`, empty-dataset (divide-by-zero guard), failure-ordering, and `AnswerFailed`-propagation tests.

Two Codex findings were deliberately left unfixed, disclosed as stated assumptions in `refresh_index`'s own docstring rather than silently ignored: `chunk_size` changing between an index's `build_index` call and a later `refresh_index` call on the same index, and `refresh_index` being handed an externally-corrupted `DocIndex` whose `content_hashes` don't match its own `chunks` — neither is a realistic path given this exercise's fixed-corpus-config premise (a `DocIndex` is only ever produced by `build_index`/`refresh_index` themselves).

**Stage 3 (Fable-model critique of that remediation, reading the actual current files and constructing its own mutants rather than trusting the write-up):** found the remediation was substantially real (independently re-verified the 32-test count and every attempt's isolation numbers, and 3 of 5 spot-checked "verified live" claims held up exactly as described) but not fully complete. Most severe: an `evaluate` that hardcodes `top_k=3` internally regardless of its own `top_k` parameter, paired with a `compare_top_k` reimplemented as an *independent* loop (never actually calling `evaluate`) rather than genuine delegation, passed all 32 tests at that point — no test called `evaluate` directly with a non-default `top_k` and checked the result would differ from the default. **Empirically confirmed live.** Also found: the no-recompute spy test (`call.args[0]`) would `IndexError` — a false *failure* on a correct implementation that calls `chunk_document` with keyword arguments instead of positionally, a real precision gap though not an exploitable bypass; and `failures` deduplication was untested, since no dataset anywhere had two cases sharing the identical query string (a `dict.fromkeys`-style dedup passed all 32 tests, confirmed live). A stale "21 tests" reference was also found in this file's own "Files" section, missed by an earlier edit pass that updated the Results table but not every mention.

**Fixed:** the spy test now handles both positional and keyword calling conventions. Two new tests added: `test_evaluate_respects_a_non_default_top_k_directly` (calls `evaluate` directly with `top_k=1` against an index where only a restricted retrieval changes the outcome, confirmed live against Fable's exact constructed mutant — fails cleanly, 1/34) and `test_evaluate_failures_includes_every_failing_case_even_duplicate_queries` (a dataset with two identical failing queries, confirmed live against a `dict.fromkeys` mutant — fails cleanly, 1/34). Test suite grew from 32 to 34. All four flawed dry-run attempts re-synced and re-verified with clean isolation against the final 34-test suite. Full regression re-run across all 25 Part 1 attempts and all 5 Module 07 attempts — zero regression. Stop condition met: this third cycle's remaining findings (the disclosed-out-of-scope items, re-confirmed as genuinely low-risk under adversarial testing) required no further code changes, only this documentation pass.

Full record in `docs/decisions.md`'s 2026-07-17 Module 08 entries.

## What this does and doesn't validate

**Validated:** the deterministic tier discriminates all four constructed flaws with clean, non-overlapping isolation after every remediation round; `check_module_08` correctly chains `check_module_07`; the checker's test-execution applies the neutral-temp-dir + expected-pass-count hardening from Module 07's own doubt-driven-development *from the start*; the checker's bypass resistance itself was adversarially tested (the fake-summary-line claim) and held.

**Not validated:** no real learner attempt exists yet, same open gap as every prior module. `score_chunk`'s deterministic keyword-overlap scoring is a coarse stand-in for real semantic retrieval (no embedding model is available in this environment) — disclosed directly in `doc_qa.py`'s own docstring rather than presented as production-grade RAG. The two disclosed-but-unfixed assumptions (stable `chunk_size`, internally-consistent input `DocIndex`) remain genuinely untested edge cases, not just documented ones — acceptable given how unrealistic both paths are for this exercise's actual scenario, but worth naming plainly rather than treating "documented" as equivalent to "resolved."
