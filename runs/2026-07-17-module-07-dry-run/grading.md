# Module 07 dry run: Designing the Solution — Architecture, Models & Context Strategy

**Date:** 2026-07-17.
**Purpose:** validate that Module 07's deterministic tier (`scripts/verify_module_07.py`) discriminates real attempts on both of its deliverables — the code artifact (`classify_ticket`) and the written artifact (a real architecture decision record) — and that this is the first module of a genuinely new shared project, `foundry`, not a continuation of `resolve`.

## A structural pivot, stated explicitly

Part 1 (Modules 01-06) built one shared project, `resolve`, incrementally. Part 2 opens on a new shared project, `foundry` (`fixtures/foundry/`) — see `fixtures/foundry/SPEC.md`'s own rationale for why `resolve` doesn't naturally stretch to cover CCAR-P's 7 Professional domains, Domain 7 (Developer Productivity & Operational Enablement) above all. This is also the first module whose deterministic gate checks two genuinely different kinds of artifact in one pass: a real, tested Python module, and a real, structurally-checked prose document (an ADR) — Part 1 never combined both in one module's gate.

**Cumulative-gate scope, stated explicitly:** `check_module_07` does not chain back through Part 1's `check_module_06`. Module 06's own hard-prerequisite framing already establishes a learner reached Module 07 having cleared all of Part 1; re-validating six already-passed `resolve` gates against an unrelated `foundry` fixture would test nothing real. `check_module_07` is a fresh starting point other Part 2 modules will chain onto, the same role `check_module_01` played for Part 1.

## The exercise's real design tension

The Helpdesk team's own first instinct — "we want something like `resolve`" — is the exercise's actual test. Most of their real ticket volume is single-turn structured classification with no genuine need for multi-step tool use or agentic autonomy; the CCAR-P skill this exercise targets (Task Statement 1.x's architectural pattern selection) is recognizing *when not to reach for the more powerful pattern*, not demonstrating that an agentic loop can be built (Part 1 already proved that). The exercise's cache-friendliness requirement (`build_triage_prompt` takes no arguments; the system prompt must be byte-identical across every call, including retries) is a second, independent real property — directly anchored to the exam guide's own Sample Question 2 (a large, repeatedly-sent static system prompt) — that a correct architectural-pattern choice alone doesn't automatically satisfy.

## Attempts constructed

1. **`correct-attempt/`** — a real single-turn `classify_ticket` (schema-validated, retries with the specific error fed back through the *turn content*, never the system prompt) plus a real 4-section ADR naming and rejecting the Helpdesk team's own proposed alternative.
2. **`no-retry-attempt/`** — `classify_ticket` makes exactly one attempt regardless of `max_retries`, raising `TriageFailed` (or returning invalid data) on the first validation failure instead of retrying.
3. **`system-prompt-mixes-ticket-attempt/`** — appends the ticket-specific text directly into the "static" system prompt before calling `model_client`, so it's no longer identical call to call — the most direct violation of the cache-friendliness requirement.
4. **`regenerates-prompt-on-retry-attempt/`** — mirrors Module 02's own retry pattern (`extract_refund_request` embeds the prior error into a freshly-regenerated prompt each attempt) applied here, where it's wrong: the system prompt changes on every retry, defeating cacheability specifically under retry pressure, a subtler and narrower violation than attempt 3's.
5. **`weak-adr-attempt/`** — `classify_ticket` is fully correct (12/12 code tests pass), but the ADR is thin: no "Alternatives Considered" section at all, and a one-line "Consequences" with no real content — passes every code test while failing the written-artifact half of the gate entirely.

## Doubt-driven-development review, before treating this module as done

Standing practice, continued from every prior module. Three stages, all with real, severe findings — the most consequential review this project has run to date, since the dominant finding wasn't scoped to Module 07 alone.

**Stage 1 (fresh Claude subagent, adversarial review of the artifact + contract, no CLAIM/reasoning shared):** 10 findings. Most severe and actionable: (1) the retry-content test (`test_classify_ticket_system_prompt_stays_identical_across_a_retry`) was satisfiable by a hardcoded "...invalid..." string via a generic OR-fallback, never requiring the *specific* validation error; (2) the documented default `max_retries=2` was never actually exercised by any test (every test passed it explicitly); (3) no test ever classified the same ticket text twice, so a memoizing implementation that skipped calling `model_client` on a repeat would pass undetected; (4) whitespace-only `detail` (`"   "`) passed `_validate`'s naive truthiness check as a valid "other" classification.

**Stage 2 (Codex cross-model, `codex exec --sandbox read-only`, same ARTIFACT + CONTRACT):** confirmed several Stage 1 findings independently and found the dominant issue of the entire review: **`run_triage_tests` (and the identically-shaped test-execution function in all five already-merged Part 1 checkers, `verify_module_02.py` through `verify_module_06.py`) ran pytest with `cwd=target`, the submission's own directory.** Since `python -m pytest` puts the current working directory at the front of `sys.path`, a submission could ship a 2-line `pytest.py` at its root (`raise SystemExit(0)`) that shadows the real installed pytest package, skipping the entire canonical test file without ever executing it — passing the whole deterministic gate regardless of how broken the actual implementation was. **Empirically confirmed live**, not just by inspection: a copy of Module 04's `correct-attempt` with its `agent.py` swapped for the known-broken `no-hook-attempt` version (missing the safety hook Module 04's own earlier doubt-driven-development had found and fixed), plus a shadow `pytest.py`, produced a clean `Deterministic tier: PASS`.

**Fix applied:** all six checkers' test-execution functions changed to copy the submission's `src/` directory into a fresh, neutral temp directory and run pytest with `cwd` set to *that* directory — never `cwd=target`. Re-verified: the same Module 04 bypass attempt now correctly fails; all 25 Part 1 dry-run attempts and all 5 Module 07 attempts reproduce their exact prior pass/fail pattern (zero regression).

**Stage 3 (Fable-model critique of the remediation itself, reading actual current files, not a summary):** found the `cwd` fix closed the specific *shadow-module* vector but not the underlying *class* of bug — a submission's own code still executes during pytest's collection step (importing `src/ticket_triage.py` is exactly that), so a submission with `os._exit(0)` at module import time kills the whole pytest subprocess with return code 0 *before a single test runs*, and the checker's `returncode == 0` check alone read that as a clean pass. **Empirically confirmed live** the same way: `correct-attempt` with `src/ticket_triage.py` replaced by `import os; os._exit(0)` produced `Deterministic tier: PASS`. Fable also found: a hardcoded `for _ in range(3):` (ignoring `max_retries` except for the negative-value guard) would pass all 19 tests at that point; the ADR fence-stripping mitigation (added mid-review, see below) could falsely delete real ADR content across an unclosed code fence; and several docstrings referenced a "docs/decisions.md 2026-07-17 entry" that didn't exist yet at review time.

**Fix applied:** all six checkers now additionally require the pytest summary output itself report the full expected test count passed (computed by counting `^def test_` in the canonical test file, checked as the literal substring `f"{expected} passed"` in stdout+stderr) — not just a zero exit code. Re-verified live: the `os._exit(0)` bypass now correctly fails on Module 04's checker and Module 07's; `correct-attempt` still passes. Also fixed: a 20th test (`test_classify_ticket_respects_a_non_default_max_retries`, using `max_retries=4` instead of the ever-present 2, expecting exactly 5 calls) to distinguish "respects the argument" from "hardcoded to match the default"; the ADR fence-strip regex now only fires when `` ``` `` markers appear in balanced (even) count, so an unclosed fence in an otherwise-valid ADR can no longer make the regex delete real section content — the remaining gap (a fake heading inside a non-```` ``` ```` fence, e.g. `~~~` or indented code) is disclosed as a known limit of a coarse regex-based proxy, not silently assumed solved.

**Stop condition met:** the third cycle (Fable's critique of the second remediation) surfaced only already-anticipated documentation staleness (this file, `docs/decisions.md`'s test-count reference) — closed by this same documentation pass, not a further code cycle.

**Also fixed from Stage 1** (independent of the systemic finding): the retry-content test now requires the literal invalid category value (`"not_a_real_category"`), not a generic "invalid" placeholder, and additionally checks the original ticket text is still present in the retry's turn content; the default-`max_retries` gap closed with a dedicated test; the memoization gap closed with `test_classify_ticket_calls_model_client_again_for_a_repeated_identical_ticket`; the whitespace-detail gap closed in `_validate` (`.strip()`) with a dedicated test; two more Stage 1/Stage 2 findings addressed defensively: non-dict `model_client` responses are now treated as a validation failure and retried rather than crashing (`test_classify_ticket_retries_past_a_non_dict_model_response`), and `max_retries < 0` now raises `ValueError` immediately, before any `model_client` call, instead of falling through to an `IndexError` (`test_classify_ticket_raises_value_error_for_negative_max_retries`); `TICKET_CATEGORIES` converted from a list to a tuple, matching `fixtures/resolve/src/tool_errors.py`'s `ERROR_CATEGORIES` precedent; `build_triage_prompt` given a real content floor (`test_build_triage_prompt_mentions_all_five_categories`) since a prompt as thin as `return "classify"` previously satisfied every test. Test suite grew from 12 to 20. All four flawed dry-run attempts and `weak-adr-attempt` were re-synced with the shared base fixes (tuple, non-dict guard, whitespace fix, negative-`max_retries` guard) so each still fails the new suite for exactly its own original, specific, intended flaw.

## What this does and doesn't validate

**Validated:** the deterministic tier discriminates all four constructed flaws correctly and with clean isolation after every remediation round, including the subtle distinction between a design that never protects cache-friendliness (attempt 3) and one that only breaks it under retry pressure (attempt 4); the ADR structural check is independent of the code check, evidenced directly by `weak-adr-attempt` passing all 20 code tests while failing the gate overall; the new project's cumulative-gate scope decision (no chain-back into Part 1) works as designed; the checker can no longer be defeated by a submission that prevents its own test suite from actually running (two independent bypass classes, both empirically confirmed and closed); all 25 Part 1 dry-run attempts still reproduce their exact original pass/fail pattern after the same fix was applied to their own checkers.

**Not validated:** no real learner attempt exists yet, same open gap as every prior module. The ADR check's "Alternatives Considered mentions resolve/agentic" keyword check is a coarse, deterministic proxy for "actually addresses the team's real alternative" — a technically-passing ADR that mentions "agentic" in an unrelated sentence would pass the deterministic check while still being a weak analysis; this is exactly the kind of gap this module's own conceptual rubric criterion (quality of the trade-off reasoning, not just section presence) exists to catch. The ADR fence-strip mitigation still can't catch a fake heading hidden inside a non-```` ``` ```` fenced block (e.g. `~~~`) — disclosed, not fixed, given the cost of a full markdown parser against the value at this scale. No test exercises the pytest-summary-count check against a submission whose test file itself is somehow correct but slow/flaky enough to produce a non-standard pytest summary line format (an extremely unlikely real-world case, not pursued).
