# Shipping Readiness Review: Platform Docs Q&A System

## Failure Modes

Three real failure modes identified before this system goes to real internal users. First, stale retrieval: `refresh_index`'s own diagnose-and-fix exercise (Module 08) exists because a runbook revision that isn't correctly re-indexed leaves the system confidently answering from a superseded procedure ‚Äî fixed, but worth naming here as a class of failure this system is structurally exposed to any time the corpus changes faster than refresh runs. Second, retrieval quality: `score_chunk`'s deterministic keyword scoring is a coarse stand-in for real semantic retrieval, so a query using different vocabulary than the indexed document can retrieve nothing relevant even when a relevant document exists ‚Äî `answer_question` fails closed in this case (`AnswerFailed`, no fabricated answer), which is the correct failure mode, but it does mean real users will occasionally get "no answer" for questions a human would answer easily. Third, and the one this module's own exercise addresses directly: sensitive-content exposure, where a retrieved chunk contains security-sensitive material (credentials, SSNs referenced in old incident writeups) and an ungoverned system would hand that straight to a model call and back out to whichever employee asked.

## Compliance Requirement and Architectural Consequence

The Platform Docs team's own corpus includes internal security runbooks and credential-rotation procedures that reference genuinely sensitive content as a byproduct of documenting real incidents. Foundry's own internal data-handling policy requires that sensitive content never be forwarded to a third-party model call without a human decision in the loop first ‚Äî this is a real, binding requirement, not a nice-to-have. The architectural consequence is direct: `doc_qa.answer_question` alone is not sufficient to ship. `governance.py`'s `answer_question_with_governance` sits in front of it and inspects retrieved chunks for sensitive markers *before* any call to `model_client`, withholding the answer entirely rather than allowing the model to see flagged content. This is a real design change forced by the compliance requirement, not a policy document sitting alongside unchanged code.

## Human-in-the-Loop Checkpoint

`answer_question_with_governance` is the checkpoint: any query whose retrieved context contains a sensitive marker returns `requires_human_review=True` and `answer=None`, with `model_client` never called. Release requires `approve_and_release`, which takes a real, non-empty `approver_id` ‚Äî a genuine audit trail of who approved exposing flagged content, not just that someone clicked a button. An empty or whitespace-only `approver_id` raises `ValueError` before any model call, so there is no path that accidentally releases sensitive content without an attributed approval.

## Stakeholder Summary

For the Platform Docs team and their own leadership: this system answers employee questions from your internal documentation, grounded only in retrieved content (never fabricated), and automatically withholds any answer that would have surfaced credential- or PII-adjacent material pending a named person's explicit sign-off. The trade-off worth knowing about: a small number of legitimate questions touching sensitive runbooks will now require a short human-review step instead of an instant answer ‚Äî the cost of the sensitivity gate is added latency on a minority of queries, not reduced coverage on the rest. Recommended before general rollout: `refresh_index` needs to run on a schedule matched to how often the underlying docs actually change, since staleness (see Failure Modes) is a real risk this review doesn't eliminate, only makes diagnosable.
