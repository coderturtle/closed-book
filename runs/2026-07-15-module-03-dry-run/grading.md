**Purpose:** validate that Module 03's deterministic tier (`scripts/verify_module_03.py`, chaining Module 02's checker, which chains Module 01's) discriminates real attempts, and that the rubric's split between deterministic and conceptual criteria is drawn correctly.

## A real bug found before any attempt was graded

The first version of every tool file put its description in the **module** docstring (the `"""..."""` at the top of the file), not the **function's own** `__doc__`. `get_customer.__doc__` and `lookup_order.__doc__` came back empty strings, not the intended text — a real bug: real MCP tooling derives a tool's description from the function's own docstring. Fixed by moving every tool's description into the function body, in both the shipped stubs and this dry run's reference implementation, before any attempt was constructed against the old (broken) shape.

## 2026-07-15 update: a doubt-driven-development review, and what it changed

Before this module was treated as done, the same process used for Modules 01-02 ran again: a fresh-context Claude subagent, Codex CLI (with live repository exploration), and a Fable-model replan. Two structural issues dominated, more severe than either prior module's review:

1. **The cumulative-gate chain was already broken going into Module 04.** `check_module_01` returned a bare `CheckResult`; `check_module_02` (added during Module 02's own remediation) returned `tuple[CheckResult, str]`; Module 03's first draft never exposed an importable `check_module_03` at all. Fixed: unified every `check_module_NN(target) -> CheckResult` on one shape (`CheckResult` gained an `output: str` field), each module's `main()` now calls its own `check_module_NN` instead of duplicating its logic, and `check_module_03` was added. Documented as a stated interface convention in `fixtures/resolve/SPEC.md`, not left implicit.
2. **The checker trusted a learner-editable test file.** `verify_module_02.py` and the first version of `verify_module_03.py` both ran `tests/test_extraction.py`/`tests/test_tools.py` *from the submitted target path* — a submission could weaken or delete the test file entirely and pass trivially; the "do not edit this file" notice at the top of each test file was prose, not enforcement. Fixed: both checkers now copy the *repo's own canonical* test file to a temp location and run that copy, with `cwd` set to the submission so imports still resolve against it. Verified empirically both that legitimate submissions still pass unchanged and that a tampered `tests/test_tools.py` (replaced with a single no-op test) no longer helps a broken submission — see Results, below.

Other real findings closed in the same pass: `process_refund`'s docstring (the module's flagship safety-critical tool) had no deterministic keyword test at all, even though the rubric claimed all four tools were checked — only `get_customer` and `lookup_order` actually were; the "re-verify BEFORE doing anything else" ordering claim was asserted in prose but untestable, since `FakeBackend` recorded no call order — fixed by adding call tracking and a test asserting `find_customer` is called before `find_order`, and that `find_order` is never called at all when the customer doesn't verify; `ERROR_CATEGORIES` had no documented semantics (a test hard-locked `errorCategory=="validation"` for an amount-exceeds-refundable case a learner reasoning from the docstrings alone couldn't have predicted) — fixed with an explicit definition of all four categories in `tool_errors.py`, and the list became a tuple; `escalate_to_human` took an injected `backend` with no stated purpose — fixed by documenting it as this project's deliberate fail-open path (verified by a new test asserting an *empty* backend still succeeds given a complete summary, the opposite assertion from `process_refund`'s); `lookup_order`'s docstring contained meta-commentary referencing "the exam guide's own Sample Question 2" that a real production tool description would never contain — stripped; the leak-prevention test only checked that a string was absent, not that the response was actually indistinguishable from a genuinely-nonexistent order — tightened to an equality assertion; the README's takeaway promised a `.mcp.json` template the exercise doesn't actually produce — reworded to state that honestly; two checkpoint answers (Q8's forced-`tool_choice` reset, Q12's Read+Write-vs-unique-anchor tradeoff) gained precision caveats.

**Explicitly deferred, not fixed, and stated as a deferral rather than left as a silent gap:** `process_refund`'s "success" response has no real persistence, decrement of `refundable_cents`, or idempotency protection — a solution can return the same success shape every time without anything actually changing state. `backend.py`'s docstring now states directly: this module verifies a refund *decision*; refund *execution* (persistence, idempotency) is explicitly Module 04's exercise, once a stateful backend exists. Named this way because Module 04's own SPEC.md capability line already commits to "the coordinator agent... with a programmatic hook enforcing verify-before-refund" — execution semantics belong there, not duplicated here.

Full reconciliation: `docs/decisions.md`'s 2026-07-15 Module 03 entries.

## Attempts constructed

1. **`correct-attempt/`** — real implementations of all four tools, real prompt/docstring content throughout, `process_refund` independently re-verifies `customer_id` before looking up the order (checked by call order, not just final result).
2. **`trusts-caller-attempt/`** — `process_refund` skips its own `backend.find_customer` re-verification entirely, reasoning "Module 04's hook already checks this."
3. **`weak-docstring-attempt/`** — all four tools behave correctly; `get_customer`/`lookup_order`'s docstrings are minimal ("Gets a customer. Takes an identifier which could be an email, phone, or account thing.") — technically containing the required disambiguating keywords, giving an agent no real boundary reasoning.

## Results (final, 20 tests)

|---|---|---|---|---|
| correct | PASS | PASS | PASS | **PASS** (20/20) |

Cumulative-gate isolation and tampering checks, both re-confirmed after remediation:
- Running against `fixtures/resolve/` directly (no Module 01 config) fails at the cumulative-gate step, before any Module 03 test runs.
- Copying `trusts-caller-attempt` and replacing its own `tests/test_tools.py` with a single no-op test **does not help it pass** — the checker runs the repo's canonical test file regardless, correctly reporting the same 2 failures.

## Finding

`trusts-caller-attempt` fails narrowly and precisely on the two tests that exist specifically to catch it. `weak-docstring-attempt` passes every deterministic test while giving an agent nothing usable to disambiguate `get_customer` from `lookup_order` beyond keyword-matching — exactly the case rubric criterion 3 exists to catch, constructed before shipping rather than found afterward. The doubt-driven-development review found something more fundamental than either constructed attempt could: the grading infrastructure itself (the chain into Module 04, and the trust boundary around test files) had real, structural gaps that no naive-vs-correct attempt comparison would ever surface, because both classes of bug are about the *checker*, not about any given submission.

## What this does and doesn't validate

**Validated:** the deterministic tier discriminates real behavioral attempts correctly; the cumulative gate chains through both prior modules and blocks correctly when they're not satisfied; the checker is no longer fooled by a submission editing its own test file; rubric criterion 3's necessity is directly evidenced by `weak-docstring-attempt`.

**Not validated:** rubric criterion 4 (defense-in-depth reasoning documented as deliberate) has no attempt isolating it specifically from the now-deterministic ordering test. No real learner attempt exists yet, same open gap as every prior module.
