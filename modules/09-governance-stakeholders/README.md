# Module 09: Shipping Responsibly — Governance, Stakeholders & Team Enablement

## The question this module answers

How do you ship this responsibly, and keep a team productive building on it, once "does it work" has already been answered?

## Where it sits in the arc

Ninth module. Hard prerequisite: [Module 08, Building and Proving It](../08-integration-evaluation/README.md) — governance and delivery apply to a system that's been built and evaluated; there's nothing real to govern or hand off until Module 08's evaluation exists. Next: [Module 10, Professional Capstone](../10-professional-capstone/README.md). See [`modules/README.md`](../README.md).

## Learning objectives (placeholder — finalized when content is authored)

- Implement guardrails and human-in-the-loop validation appropriate to a stated risk level.
- Identify a real compliance requirement (e.g. GDPR, HIPAA) that changes an architectural decision, not just a policy document.
- Run a structured discovery/requirements-gathering pass and communicate a resulting architectural trade-off to a non-technical stakeholder.
- Configure Claude Code and AI-assisted tooling for team-wide developer productivity, not just personal use.

## Exercise material to draw from (not a spec — Coachgremlin authors the real exercise later)

Real material this module's exercise should be built from: CCAR-P Exam Guide Domain 5 (Governance, Safety & Risk Management, 14%), Domain 6 (Stakeholder Communication & Lifecycle Management, 14%), and Domain 7 (Developer Productivity & Operational Enablement, 7%) — their full listed objectives (guardrails, risk/failure-mode identification, human-in-the-loop strategy, regulatory compliance, ethical AI considerations, structured discovery, stakeholder communication/SLA management, architecture documentation, lifecycle-phase support, team tooling configuration, debugging/operational support). See [`docs/workshop-design.md`](../../docs/workshop-design.md).

## Required gate (placeholder — shape decided now, real rubric written later)

- **Deterministic tier (hands-on, with Claude Code):** a real governance/risk review of Module 08's system (documented failure modes, a stated compliance requirement and its architectural consequence, a human-in-the-loop checkpoint actually wired in) plus a written stakeholder-facing architecture summary.
- **Exam-condition tier (closed-book, without Claude Code):** a timed multiple-choice checkpoint against Domains 5, 6, and 7's combined task statements. Default format: 10–15 questions, 15–20 minutes, 80% to pass.

## Takeaway

A governance/risk checklist (compliance, human-in-the-loop, stakeholder sign-off) — reusable on the learner's next real system, built from a real reviewed system rather than a generic template. Packaged by Coachgremlin once the rubric is met.

## Stop condition (placeholder)

The learner's governance review is real and specific to the system reviewed, the stakeholder summary is written and defensible, and Coachgremlin confirms the closed-book checkpoint was passed at 80%+ under real exam conditions.
