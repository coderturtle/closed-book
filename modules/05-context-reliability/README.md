## Exercise: case facts and the escalation decision

Runs against `fixtures/resolve/`, continuing the shared project. Implement two functions in `fixtures/resolve/src/context.py`, both currently `raise NotImplementedError`, both already carrying a real docstring stating their contract. `CaseFact` and `CaseFacts` (the data structures) are already provided, not part of the exercise:

- **`update_case_facts`** — folds one tool call's result into a persistent `CaseFacts` record. A failed result (`{"error": True, ...}`) increments `error_count` and updates nothing else — a failed call establishes no fact, the same "success, not attempt, is what counts" discipline Module 04's own `verified_customer_id` already applies. A successful result updates the field it maps to (`get_customer` → `customer_id`, `lookup_order` → `order_id`, `process_refund` → `refund_amount_cents`), reading the value from the *result*, never from the request, for **every** mapped field alike — the same "trust the result, not the ask" lesson Module 04's own doubt-driven-development review had to add a test for the first time; this module's own dry run needed a second round to catch that its first test suite only enforced this for one field, not all three (see below). If a field already holds a *different* value, that's a conflict: append `(field_name, old_fact, new_fact)` to `case_facts.conflicts` — recording the disagreement, not silently picking a side — while still updating the field's current value to the newer result. Must not mutate the `case_facts` argument in place, including its *nested* mutable state — a shallow copy that reuses the original's `conflicts` list object still corrupts the caller's original the moment a later conflict is appended, even though every scalar field looks untouched.
- **`should_escalate`** — decides whether a session should hand off to a human right now, checking exactly three structured signals in order: (1) any unresolved entry in `case_facts.conflicts`, (2) `case_facts.error_count >= 3`, (3) `iterations_used >= max_iterations - 1`. Returns a specific reason string for whichever applies first, or `None`. Deliberately has no `confidence` parameter in its signature — Task Statement 5.5's own core lesson is that a model's self-reported confidence or sentiment is an unreliable proxy; this function can't lean on one because it was never given access to one. This guarantee is about the function's actual signature, not just its current body — an unused, defaulted-to-`None` `confidence` kwarg would reopen the door the design closes, even if nothing calls it with a real value yet.

**A relationship to Module 04 worth stating explicitly, not left implicit:** Module 04's `SessionState.verified_customer_id` is last-verified-wins by design — a session enforces one active verified customer for authorization purposes, and overwriting is exactly correct there. This module's `CaseFacts.conflicts` does the opposite for a different reason: it's a reliability/audit concern, not a real-time authorization one, so silently keeping only the newest value would hide a real signal a human reviewing the case needs to see. Both behaviors are correct for what each is for — a learner who's just internalized Module 04's "overwrite is correct" lesson could reasonably (and wrongly) import that assumption here.

A real, provided test suite (`fixtures/resolve/tests/test_context.py`, 30 tests) checks, among other things: facts are recorded with correct provenance (`source_tool`) from each of the three mapped tools; the request-vs-result discipline holds for `lookup_order` and `process_refund` alike, not just one of them; a failed call increments `error_count`, records no fact, and does not discard facts an earlier, unrelated call already established; the original `CaseFacts` passed in is never mutated, including its `conflicts` list specifically (not just its scalar fields); a changed fact is recorded as a conflict with both sourced values preserved for `customer_id`, `order_id`, and `refund_amount_cents` alike, while an *unchanged* re-report of the same value is not a conflict; independent conflicts on different fields both survive in `conflicts` rather than overwriting each other; `should_escalate`'s own signature is checked directly (exactly three parameters, no confidence-shaped extra) alongside its behavior: conflicts before the error threshold before the iteration threshold, a distinct reason string for each, and the iteration boundary checked exactly, not just "near" vs. "well before."

**Before trusting a green checker as proof you're done:** this module went through three rounds of scrutiny before shipping, all real. First, its own dry run (`runs/2026-07-16-module-05-dry-run/grading.md`) constructed four flawed attempts plus a proactively-built conceptually-weak one before any real learner touched it, continuing the discipline Module 04 established. One of them (`confidence-proxy-attempt`) is worth reading closely even if your own implementation passes every test: it checks only iteration count, ignoring `case_facts` entirely. Second, a doubt-driven-development review (a fresh Claude subagent + Codex, both constructing and running live counter-implementations, not just reasoning about them) found the *first* version of `test_context.py` had a real, systemic gap: it tested `customer_id` far more rigorously than `order_id` or `refund_amount_cents` — a request-vs-result mixup, a hollow conflict entry, or a silently-overwritten conflict on either of those two fields would all have passed 20/20. It also found a shallow-copy implementation of the non-mutation contract would pass every test while still corrupting the caller's original `conflicts` list, and that `should_escalate`'s "no confidence parameter" guarantee was asserted in prose but never actually checked against the function's real signature. Third, a Fable-model critique of that remediation found the "symmetric coverage" fix was itself still asymmetric — conflict-preservation tests for `order_id`/`refund_amount_cents` checked values but not `source_tool`, a failed `lookup_order` could still leak a fact from its request args (the same key-collision the success-path fix had just closed, reopened on the error path), and refund conflicts had no most-recent-wins or no-conflict-on-repeat test at all. Every fix across all three rounds was verified against a constructed counter-implementation actually run against the checker, not just re-read for plausibility — see `docs/decisions.md`'s 2026-07-16 Module 05 doubt-driven-development entries.
