Module 07 opened Part 2 with the IT Helpdesk team wanting something more powerful than their problem actually needed. Module 10, the last module in this workshop's entire arc, ends it by pointing the exact opposite pressure at the exact same system. A VP of Engineering looks at what got built across Modules 07 through 09 — `ticket_triage`, `doc_qa` with its evaluation harness, `governance` wrapping the whole thing — and asks the question a real budget review would actually ask: why three separate small systems instead of one general-purpose internal assistant? Fewer codebases, fewer onboarding surfaces, less duplicated retrieval and model-client plumbing solved three separate times. It's not a strawman. It's the same discipline this workshop's own tool-design and architecture modules teach, aimed back at the workshop's own creation.

There's no new source file in this module, and that's a deliberate break from every prior Part 2 module's shape. Modules 07 through 09 each paired a code deliverable with a prose one. This one is purely written, because the actual skill being tested is synthesis, not new build: a real defense of `ticket_triage`, `doc_qa`, and `governance` staying genuinely separate, grounded in what the three earlier modules actually established about each one's distinct risk profile — not "different systems solve different problems," which is true of almost anything and defends nothing specific. The defense has to name the real systems, engage the VP's cost argument honestly before countering it, and end with an actual falsifiable condition under which consolidating would become the *right* call, not just the cheaper one. Budget pressure alone doesn't count. A real convergence in problem shape does.

The closed-book side scales up the same way Module 06 did for Foundations: instead of this workshop's usual ten-to-fifteen-question checkpoint, a full 42-question, six-scenario mock exam matching CCAR-P's own real structure, drawn four scenarios at a time, 720 out of 1000 to pass. New for this one specifically: two questions in multiple-response format, select-all-that-apply with exactly two correct answers, because the real exam uses that format and nothing before this module needed it. Writing forty single-answer questions and getting an even split of correct letters felt like a solved problem after Module 06 already found and fixed the "every answer is A" version of this bug. It wasn't solved. I found a new way to make the same mistake before I'd finished writing the answer key, and a second way after I thought I'd fixed the first one. That's genuinely its own story, and it's the one that closes this workshop out.
