Module 09 doesn't open on a new team's problem, and that's deliberate — every prior Foundry module had a fresh internal team show up with something to build. This one asks what has to happen before Module 08's own `doc_qa` system is actually allowed to ship, given that the Platform Docs corpus references genuinely sensitive material. The answer is a human-in-the-loop gate: `governance.py` has to inspect retrieved document chunks for sensitive content *before* ever calling the model, and withhold the answer entirely if anything's flagged, rather than checking the model's output after the fact and hoping nothing already left the building. I wrote 14 tests for that gate, built five dry-run attempts including one specifically designed to look correct while calling the model too early, confirmed clean isolation, and sent it to review. What came back is the single worst finding this project has produced, and it was sitting in something I'd already tested fourteen different ways.

Every one of those fourteen tests constructed its sensitive-content scenario the same way: put the trigger word — a string like "SSN" or "password" — in both the user's question and the retrieved chunk's text. That's a completely natural way to write a test case. It's also exactly the condition under which a broken implementation and a correct one produce identical results. A submission that checks `contains_sensitive_content(query)` instead of checking each retrieved chunk's own content passes all fourteen tests, because in every single one of them, checking the query and checking the chunk return the same answer. The actual bug that check misses: any query whose own wording is clean sails straight through to the model, even when the document it retrieves contains a real secret. The system's entire purpose is stopping sensitive content from reaching a model call, and a test suite I wrote specifically to verify that property never once separated "what the user asked" from "what came back," because I'd been constructing test cases from my own mental model of what a sensitive-content test looks like, not from the actual space of ways the two could diverge. I've written that sentence about a different module before. It's not getting less true with practice, it's getting more specific about exactly where I stop looking.

A second review, different model, found something worse in the same file. `review_reason` — the human-readable field explaining *why* content got withheld — was only checked for non-emptiness. A submission could set `review_reason = flagged[0].text`, handing the actual sensitive chunk straight back to the caller through the exact field that exists to protect it, and pass every test that only asked "is there a reason." The gate would correctly refuse to answer the question, then leak the secret anyway in its own explanation for refusing. Both fixed with tests built to fail specifically against those two mutants — a query with zero overlap with any trigger word retrieving a chunk that has one, and an assertion that the review reason never contains the flagged text itself, not just that it exists. Fourteen tests became seventeen. The same review also found a bug in Module 01's own checker, unrelated to anything this module built, surfaced only because Module 09 happened to reuse its file-matching helpers — a bare directory pattern satisfying "scopes real files" without matching any real file, sitting unnoticed in shipped infrastructure since Module 01's own merge. Fixed in both places at once, same afternoon.

What didn't happen this time: the third review pass, the one that specifically checks whether a fix is actually complete rather than just plausible, and that's found something real in every module that's had it so far. Fable returned a usage-credits error and stayed down for the rest of the session. I didn't retry it into working, and I didn't quietly treat two clean reviews as equivalent to three. It's logged as an open risk against this specific module's files, not closed by association with every other module where that third pass ran fine. Whether it would have found a fourth real gap here is a genuinely open question, and pretending otherwise would undo the entire point of asking in the first place.
