Before any review touched Module 10's mock exam, I found two bugs in it myself, which is the outcome this whole process is actually supposed to produce and rarely gets credit for when it works. Thirty-nine of forty single-answer questions had "B" as the correct letter — Module 06's own "every answer is A" bug, wearing a different letter, in a document I'd written specifically knowing that bug existed. And three questions I'd tagged as testing architectural pattern selection were actually testing risk and failure-mode identification, a skill this workshop had already established belongs to a different domain three modules earlier. Both caught with `grep` before any external reviewer saw a single line. I fixed the letter distribution with a script that moved each correct answer's text to a new position across the forty questions. That script is where the real story starts, because it broke something I didn't notice until I went looking for it on purpose.

The regex I used to find and remap distractor letters inside each answer's explanation text was too broad. It matched the letters it was supposed to remap, and it also matched the English word "A" starting a sentence, and it matched "A/B testing" sitting in the middle of one explanation as an ordinary technical term, scrambling it into "B/A testing." A first-pass review caught three of the four corrupted lines. It didn't catch the fourth, because "A/B testing" doesn't look like corruption unless you already know to distrust it — I found that one myself, by grepping for the literal string "A/B" after noticing the pattern in the other three. The file had never been committed, so there was no git history to revert to. I rebuilt the pre-shuffle content from what I'd already written earlier in the same conversation, then wrote a narrower version of the same script — one that only ever touches the parenthetical explanation after a bolded answer letter, never the sentence that states what the answer actually is. Corruption like that doesn't happen from writing a shuffle script carelessly. It happens from writing one that's *almost* precise enough, which is a worse trap because it passes a casual look.

What followed was three full review cycles, and each one found something real that the one before it had missed, in a pattern I'm starting to trust more than any single review no matter how careful. The first, a fresh model reading only the finished artifact, found two ways to game the deterministic checker with hollow prose and confirmed one finding I'd half-suspected already wasn't real — a claimed domain mislabel that turned out to match this workshop's own established precedent from three modules earlier, checked against the actual file rather than taken on the reviewer's word. The second, a different model entirely, read my already-reshuffled answer key and found that the forty correct letters formed an exact repeating cycle — A, B, C, D, A, B, C, D, straight through — which technically satisfies "even distribution" while being fully predictable by position alone. Worse than the bug I'd caught myself, in the exact same file, produced by the very fix meant to close it. The third pass did something none of the first two had: it read the *fix* for the second finding, not the original artifact, and found that my new checker logic — built to stop a hollow essay from gaming a "does this defense engage real risk vocabulary" check — would also fail a genuinely honest submission that describes a real safety guarantee in ordinary negative language, sentences like "sensitive content never reaches the model call." A check built to catch bad-faith writing was about to punish good-faith writing for using the word "never," which is a worse failure mode than the gap it was built to close, because it actively penalizes the correct answer instead of merely tolerating a wrong one.

That same pass surfaced something none of this session's own mistakes caused: every distractor across all forty-two questions used words like "always," "never," or "exclusively," while every correct answer was the longer, hedged, trade-off-shaped option — a pattern old enough that it had survived through this module's original authoring untouched by anything I'd fixed so far. Closing it meant rewriting roughly a hundred and twenty-five individual wrong answers by hand, checking each one against its own explanation to make sure the reason it's wrong didn't change even though the words did. I asked whether that was worth doing at this scale rather than just writing it down as a known limitation, which is what this project does with plenty of smaller gaps. The answer was to do it anyway, which is most of what took the rest of the session. Ten modules, four constructed dry-run attempts per module on average, a project's worth of checkers that have each been wrong at least once and gotten caught. This is where the workshop's content ends. What it hasn't done yet, not once, across any of it, is teach a real person who isn't me — and that's the actual test this whole thing has been building toward.
