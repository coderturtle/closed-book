Module 07 got the same treatment every module before it did: build it, dry-run it against constructed attempts until the checker discriminates cleanly, then hand the actual artifact and contract — stripped of my own reasoning, so it can't just agree with me — to a fresh-context reviewer and ask it to find what's wrong. That review came back with ten real findings about the exercise itself, a gameable retry test, an unexercised default, the usual shape of thing this process has caught in every module so far. Then I offered a second opinion from a different model, and that one didn't find a tenth finding. It found the checker.

`scripts/verify_module_07.py`, like every checker before it, runs the submission's code against a canonical test file by shelling out to `python -m pytest` with the working directory set to the submission's own folder. That's deliberate — it's how the test file resolves `from src.ticket_triage import ...` against whatever the learner wrote. What I hadn't accounted for is that `python -m pytest` also puts the current working directory at the front of Python's import search path. A submission that ships a two-line file named `pytest.py` — `import sys; raise SystemExit(0)` — gets found before the real, installed pytest package, and the entire test suite never runs. Zero code executed, exit code zero, checker says pass. I didn't take that on faith. I took Module 04's own hook implementation with the safety check deliberately removed — a real, previously-found bug, not a hypothetical — dropped the two-line file next to it, and ran it against the actual checker. Clean pass. Then I grepped for the same pattern across the other five checkers already merged to main. All five had it. This had been sitting in shipped, trusted infrastructure since each module's own original merge date, days earlier, and nothing in five separate rounds of review across five separate modules had caught it, because none of those reviews had reason to doubt the checker itself.

Fixed it the obvious way: copy the submission's source into a fresh, neutral temp directory instead of running in place, so nothing the submission ships can ever land on the import path ahead of the real tools. Reran the Module 04 attack, confirmed it now failed correctly. Reran all 25 Part 1 dry-run attempts to make sure nothing legitimate broke. Six files fixed, one afternoon, standard remediation loop. I could have stopped there.

I didn't, because the same discipline that flagged Module 04's hook bug applies to a fix as much as to a first draft: check the fix instead of trusting it. A third pass, reading the actual current files rather than a summary of what I'd claimed to do, found that the neutral-directory fix closed one specific attack and not the underlying problem. A submission's code still runs during pytest's own collection step — importing the file is exactly that — so a submission that calls `os._exit(0)` at the top of the module kills the whole subprocess with exit code zero before a single test executes, before pytest even gets to print anything. No shadow file required this time, no import-path trick, just an abrupt exit at the worst possible moment. Confirmed live again, same method: real broken code, this new trick layered on top, checker says pass. The actual fix had to be a different kind of check entirely — not "did the process exit cleanly," but "did pytest's own summary report the number of tests I expected to run," computed by counting the real test functions in the canonical file and requiring that exact count show up in the captured output. `os._exit` skips flushing, so a fake printed summary doesn't even survive the crash. Confirmed that too, deliberately, before trusting it.

I don't have evidence either bypass was ever used against this project for real — it's a small, mostly self-directed workshop right now, not something with outside pressure on it yet. But it was live, mergeable, shippable code for days, and the thing that caught it wasn't a smarter first draft. It was refusing to let "I fixed it" be the last word on a fix, twice in the same afternoon.
