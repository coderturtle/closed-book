`resolve`'s own spec had an open question sitting in it since Part 1 started: what does Part 2 build on? I'd left it open on purpose rather than guess, and closing it turned out to matter more than I'd assumed going in. The honest case for just continuing `resolve` was real — one shared codebase, no new scaffolding, momentum. But CCAR-P's seventh domain is Developer Productivity and Operational Enablement, and there is no honest way to hang that domain off a single customer-support agent without inventing a scenario that doesn't fit anything `resolve` actually is. Asked the question straight rather than deciding it myself: new shared system, not a continuation. Second question, same treatment: what should it be? An internal AI platform team's own product, not a scaled-up SaaS thing or a scaled-up `resolve`. Named it Foundry — the team and tooling that lets other engineering teams inside a company adopt Claude Code and the Agent SDK safely, which means its whole premise is serving a different internal team with a genuinely different problem every module. That's what `resolve` couldn't do by construction.

Module 07's problem, and Foundry's first, came from the IT Helpdesk team: roughly 4,000 tickets a month, mostly password resets and VPN access and software requests, and their own first instinct on hearing about `resolve` was "we want that, but for tickets." A full agentic loop, tools that look up account status, multi-turn autonomy. That instinct is the exercise. Most of those tickets are single-turn structured classification with no real need for multi-step tool use, memory across turns, or autonomy of any kind — the CCAR-P skill under test isn't proving an agentic loop can be built, Part 1 already proved that six times over. It's recognizing when a well-scoped prompt beats the more powerful pattern, and being able to say why to the team that asked for the powerful one by name.

The part I found genuinely interesting to design was the cache-friendliness requirement, because it's a direct contradiction of something Module 02 got right. `extract_refund_request` regenerates its whole prompt on every retry, folding the validation error back into a freshly built string — correct there, because that prompt isn't sent at any volume that makes caching pay off. Here, the same move is wrong: the IT policy text and category definitions are large, static, and sent roughly 4,000 times a month, so `classify_ticket` has to keep that system prompt byte-identical across every call, including retries, and route the retry-specific error through the turn content instead. Same underlying judgment — where does per-call variation actually need to live — landing on opposite answers in two different modules. Getting a learner to notice that transfer, and not just pattern-match "Module 02 did retries this way, do that again," is the actual point.

Module 07 is also the first module where the deterministic gate checks two different kinds of thing in one pass: real code, and a real architecture decision record defending the pattern choice against the Helpdesk team's own stated alternative, with actual trade-offs named rather than asserted. A regex-based section checker is a coarse tool for grading prose, and I said so directly in the module's own rubric rather than pretending the deterministic tier catches everything — a technically-passing ADR that mentions "agentic" in one throwaway sentence would still clear the automated check. That's what the conceptual rubric is for. What happened when I put this module through the same doubt-driven-development pass that found real bugs in every Part 1 module — including one that had nothing to do with Module 07 at all — is its own story, next.
