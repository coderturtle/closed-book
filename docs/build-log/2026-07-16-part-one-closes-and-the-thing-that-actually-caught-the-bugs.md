Six modules ago I didn't have a shared project yet, just a scaffold and a naming decision. Part 1 (Architect Foundations) is `resolve` now: a customer-support resolution agent modeled on CCA-F's own Scenario 1, built one module at a time — extraction in Module 02, four MCP tools in Module 03, the agentic loop and a safety hook in Module 04, case-fact tracking and an escalation decision in Module 05, and a capstone in Module 06 that ships the whole thing pre-integrated with two real bugs already in it, for the learner to find. Six real exercises, six real deterministic checkers chaining into each other, six closed-book checkpoints, and by the end, a discipline I didn't have on day one: don't trust your own first pass, ever, even when the tests are green.

That discipline earned its keep in Module 04. The exercise is a `PreToolUse`-style hook that's supposed to block a refund until the customer has been verified in that same session. I wrote it, wrote 12 tests, constructed 5 deliberately broken dry-run attempts to prove the tests discriminate real mistakes, and everything passed clean. Then a fresh-context review — no memory of how I'd reasoned about it, just the code and the contract — found that the hook checked whether *any* `get_customer` call had succeeded in the session, never *which* customer it verified. A session that looked up customer A could refund customer B, and none of my five constructed attempts happened to probe that specific gap, because I'd been testing my own mental model of the bug, not the actual space of ways to get it wrong. Fixed with an identity-bound `verified_customer_id`, and a follow-up pass on the fix itself found the new test only caught the false-*block* direction, not the false-*allow* direction the original bug actually reintroduces. Two more findings after that, both real.

Module 05 had the same shape of gap in a different place: the test suite checked `customer_id` far more rigorously than the two other fields case facts track, `order_id` and `refund_amount_cents`, because writing thorough tests for the first field I touched felt like I'd covered the pattern. A request-vs-result mixup or a silently overwritten conflict on either of the other two fields would have sailed through 20 passing tests. And Module 06's capstone, built specifically to integrate everything Modules 02, 04, and 05 had established, turned out to have a base-design gap none of the earlier modules' own reviews had caught: a hook rejection just skipped straight past the escalation logic entirely, contradicting the fail-closed default this whole project is supposedly built on. The mock exam's own answer key was funnier and just as real — all 42 correct answers were option A, so a test-taker could score a perfect 1000 without reading a single question, and I'd written every one of those questions myself.

None of these were catastrophic in isolation. What they have in common is that a single confident pass — mine — wouldn't have found any of them, and a second pass from a genuinely different vantage (fresh context, then a second model, then a third pass specifically checking whether the *fix* was actually complete) found real, specific, fixable problems every single time, six modules running. That's not a QA nice-to-have at this point. It's the actual argument for whether an agent-native workshop can be trusted to grade itself, and it's still an open one: no real learner has attempted any of this yet, only constructed attempts I built to fail on purpose. Part 2 starts next, and it inherits this habit whether the new material wants it or not.
