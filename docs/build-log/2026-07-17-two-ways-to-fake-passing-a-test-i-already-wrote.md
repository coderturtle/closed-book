Module 07's ticket classifier didn't need retrieval, and that was the point — recognizing when the simpler pattern is correct. Module 08 needed the opposite lesson from the same team's own catalog: the Platform Docs team's problem, employees asking ad hoc questions about internal engineering docs, is genuinely a RAG problem, not one shoehorned into looking like one. `doc_qa.py` ships mostly working on purpose — chunking, indexing, retrieval, grounded answer generation, all correct as delivered — with exactly one seeded defect in `refresh_index`: it checks whether a document's `doc_id` is already present in the index, never whether the document's *content* changed. A revised runbook keeps its stale chunks forever under the same ID. That's not an invented bug. It's Anthropic's own exam guide's third sample question, built into working code instead of described in a paragraph.

The second deliverable, `evaluation.py`, ships as a stub — `evaluate` and `compare_top_k`, build from nothing, same shape as Module 07's classifier. I wrote 21 tests, built five dry-run attempts, confirmed clean isolation, and sent it to review. What came back was two different fake implementations that both cleared all 21 tests while doing none of the real work, and they failed for opposite reasons that added up to the same lesson: I'd tested outcomes, not process.

The first: a `refresh_index` that does a full rebuild on every single call, never checking whether a document's content hash actually changed. My "leaves unchanged documents untouched" test compared the *values* on a frozen dataclass before and after — and a full rebuild produces identical values every time, so the comparison passed even though real recompute happened on every document, defeating the entire point of an incremental refresh. The second, found independently: a `refresh_index` that mutates its input index in place and returns that same object. My test compared the returned index against the original *after* the call — which is tautological when they're literally the same object in memory. Both bugs are invisible if you only ever check "did the output look right." Neither is invisible if you ask "did the object I was told not to touch get touched anyway," which is a completely different question and one I hadn't thought to ask either implementation. Fixed with a real spy on the chunking function to prove unchanged docs are never recomputed, and a before/after snapshot proving the original index object survives untouched. A second review pass, from a different model, added four more: `evaluate` could skip `answer_question`'s real retrieval path entirely, `compare_top_k` could quietly evaluate a subset instead of the full dataset per k, and a shallow dict copy in `refresh_index` let old and new indexes share the same mutable chunk lists behind the scenes. Test suite: 21 to 32.

The finding I didn't fix, because it wasn't real, is the one worth writing down. That same review claimed a checker bypass — a submission printing a fake success line before killing the process, the same `os._exit` trick that broke Module 07's checker two days earlier. I didn't take that on faith either way, same as I hadn't taken the real ones on faith. Built the exact attack, ran it against the actual checker. It failed to work: pytest's own output capturing during collection swallows a `print()` before `os._exit(0)` ever gets a chance to flush it to the process's real stdout, even with an explicit flush call in the way. The bypass that worked on Module 07 doesn't transfer here, because the failure happens at a different stage of the same subprocess and pytest's own internals get in the way by accident. Checking a fix costs the same effort as checking a claim that something's broken. Both times this module, that discipline paid for itself — once by finding two real bugs disguised as passing tests, once by refusing to spend an afternoon fixing something that was never actually wrong.
